# 麦麦晚安睡眠管理

声明：本插件90%由Codex-5.5生成，本插件90%由Codex-5.5生成，本插件90%由Codex-5.5生成

这个插件会监听 Bot 自己发出的消息：当 Bot 在夜间主动说出“我睡了”“该睡了”“大家晚安”等短句时，自动进入临时睡眠状态

它也会识别用户让 Bot 睡觉的消息：

- 如果当前不在允许入睡时间内，默认会委婉回复并拦截这条消息，避免进入 Planner
- 如果当前在允许入睡时间内，只记录一次短暂的“待确认入睡”状态，然后放行给主链路判断，并给 Planner 注入一条上下文，说明当前已经符合插件配置的入睡时间
- 只有当 Bot 随后自己发出明确的入睡确认，才会真正进入睡眠
- 插件默认使用一层轻量 AI 判定器判断 Bot 是否“自己决定要睡”。判定器只接受 `SLEEP / NOT_SLEEP / UNSURE` 三种固定结果；只有判定为 `SLEEP` 才会入睡

睡眠期间会暂停：

- 新入站消息进入主处理链路
- 表达学习提取和写入
- 新触发的人物事实写回和聊天摘要写回
- Planner 响应结果和工具调用
- 睡眠后的后续出站消息

睡眠状态会默认持久化到 `data/plugins/goodnight_sleep_manager/sleep_state.json`。如果主程序、插件 Runner 或插件本身在睡眠期间重启，插件加载时会恢复尚未过期的睡眠状态；如果预计醒来时间已经过去，会自动清理状态文件。手动 `/sleep_wake` 或自然到点唤醒也会清理对应作用域的持久化状态。

为了避免别人诱导 Bot 睡觉，插件默认只接受很短的自我睡眠或群体收尾短句。单独一句“晚安”只有在前面有人于合适时间催睡、且 Bot 自己选择这样回复时才会触发；带 `@`、明显称呼别人、或像“晚安，某某”这种对个人说晚安的内容不会触发

默认可视化配置不再要求维护一堆正则。正则仍保留在源代码配置里，作为 AI 不可用或判定不确定时的高级兜底。

AI 判定只会在允许入睡时间内触发，并且有两层限流：

- 已有合理催睡 pending 时，对 Bot 随后的回复做一次语义判定
- 没有 pending 时，只对 Bot 自己发出的、包含睡眠相关词的短句做判定

因此普通聊天不会每条都额外调用模型。

## 静默入睡

“静默入睡”默认关闭。开启后，插件会在后台定期检查各睡眠作用域的两个计时器：完全安静计时，以及 Bot 无参与计时。只要其中任意一个达到配置的分钟数，就会自动进入睡眠。

这个检查不调用 LLM，也不会为了入睡额外发送“晚安”消息，只会把睡眠状态切到生效的作用域。配置了单独作息的群会按自己的群作用域和作息窗口计算；未配置单独作息的群和私聊默认也会按各自聊天流独立计算。

即使完全没有新消息，后台检查任务也会按“检查间隔秒”醒来一次，所以仍然能发现已经进入允许入睡时间。插件在窗口外会持续刷新基准时间，进入窗口后才开始累计；如果插件启动时已经在窗口内，会先从启动后的第一次检查开始计时。

完全安静计时会被任何入站消息或 Bot 出站消息刷新，适合“群里真的一片安静就睡”。无参与计时只会被 Bot 出站消息，或 Planner 的有效动作刷新；如果群里有人聊天，但 Bot 的 Planner 最终选择 `no_action`、`no_reply`、`no_react`、`no_plan`、`finish`、`wait`、`continue`，或者没有任何出站消息，就会继续累计无参与时间。

临近无参与入睡时，新入站消息会获得一次“话题判断缓冲”，让 Planner 有时间判断这个话题是否值得参与。普通消息在同一个无参与周期里只给一次缓冲，避免热闹群无限续睡；如果 Bot 真的参与了，计时和缓冲都会重置。被提及或被 `@` 时可以按配置延长缓冲，更适合“提及必回复”或 “@ 必回复”开启的场景。

已经进入睡眠后，默认不会被普通提及叫醒。需要更“浅睡”的拟人效果时，可以开启“睡眠中提及唤醒”，这样被提及或 `@` 会先唤醒，再放行这条消息进入主链路；关闭时仍然按睡眠消息拦截，并可进入睡醒回顾。

相关配置：

- `启用静默入睡`：是否允许安静到一定时间后自动睡觉
- `完全安静入睡分钟`：没有任何入站/出站消息多久后自动入睡
- `无参与入睡分钟`：Bot 连续多久没有参与话题后自动入睡
- `检查间隔秒`：后台检查最近活动时间的频率，不消耗模型 token
- `话题判断缓冲秒`：临近无参与入睡时，给新话题进入 Planner 判断的时间
- `提及延长缓冲` / `@ 延长缓冲`：直接喊她时延长临睡前判断缓冲
- `睡眠中提及唤醒`：睡着后被提及或 `@` 是否自动醒来，默认关闭
- `Planner 动作算参与`：开启后有效 Planner 动作会刷新无参与计时，`no_action`、`no_reply`、`no_react`、`no_plan`、`finish`、`wait`、`continue` 不算参与

## 分群作息

全局“作息”页仍然作为默认时间配置。需要某个群单独使用不同作息时，可以在“分群作息”里添加一条群作息覆盖，填入群号和这组时间：

- `允许入睡开始`
- `允许入睡结束`
- `目标醒来时间`
- `最短睡眠分钟`
- `最长睡眠分钟`
- `醒来随机浮动`

当消息来自该群时，分群作息的优先级高于全局作息；未命中任何群号时继续使用全局作息。用户在允许时间内催睡时，Planner 提示里也会说明当前使用的是分群配置还是全局配置。

睡眠状态也会按作用域隔离：

- 配置了单独作息的群：使用 `group:<群号>` 独立睡眠状态
- 未配置单独作息的群：默认也使用 `group:<群号>` 独立睡眠状态，但时间仍沿用全局作息
- 未配置单独作息的私聊：默认使用自己的聊天流独立睡眠状态，时间仍沿用全局作息

也就是说，默认情况下，某个群因为安静或无参与入睡，不会切断其他群正在进行的聊天。如果想恢复旧逻辑，让所有未配置分群作息的群聊和私聊共用一个全局睡眠状态，可以关闭“默认聊天流独立睡眠”。

## 睡醒回顾

“睡醒回顾”默认关闭。开启后，插件会在睡眠期间把被拦截的消息保存为本地轻量记录，并在对应作用域醒来时按群聊/私聊生成回顾文件。

回顾文件位于 `data/plugins/goodnight_sleep_manager/sleep_review/reports/`，会保存：

- 群号、群名、私聊用户、QQ 号、昵称/群名片
- 睡眠期间的轻量聊天记录
- 每个聊天流的一段简短总结和重要上下文

它不会向任何群聊或私聊补发回复，也不会对历史消息逐条响应。配置了单独作息的群会在自己的睡眠作用域醒来时单独生成回顾；未配置单独作息的群和私聊默认也按各自独立睡眠作用域回顾。关闭“默认聊天流独立睡眠”后，它们才会跟随全局睡眠作用域一起回顾。

为了控制 token，睡醒回顾页提供了成本上限配置：

- `单聊最大消息数`：每个群聊/私聊最多送入总结模型的最后 N 条消息
- `单聊最大字符数`：每个群聊/私聊最多送入总结模型的文本字符数
- `单次最大聊天流`：一次醒来最多调用模型总结的聊天流数量
- `单聊总结输出上限`：每个聊天流总结的输出 token 上限

这些限制只在“启用睡醒回顾”打开后生效；未启用时不会记录消息，也不会调用模型总结。

## 命令

- `/sleep_status`：查看当前聊天流对应作用域的睡眠状态
- `/sleep_wake`：手动唤醒当前聊天流对应作用域
- `/sleep_now`：在当前生效作息允许入睡时，记录一次管理员催睡并交给主链路判断；如果 Bot 随后自己说晚安，插件才会入睡
- `/sleep_force`：无视当前时间窗口，立即进入睡眠，适合管理员测试或紧急控制

`/sleep_now` 和 `/sleep_force` 可以通过“拦截”页里的“允许管理入睡命令”关闭。填写“管理员用户 ID”后，只有列表内用户能使用这两个命令；留空则不限制。

## 注意

当前 MaiBot 的 `maisaka.planner.before_request` hook 不允许直接中止请求，所以插件会在睡眠期间清空 Planner 工具、压缩请求内容，并在响应后清空结果。新消息入口会被拦截，因此正常情况下不会再由新消息触发 Planner

“睡眠期间暂停长期记忆自动写回”依赖主程序提供 `memory.automation.before_enqueue` hook。插件加载时会尝试检测该 hook 是否存在：若你的 MaiBot 版本尚未合并包含该 hook 的主程序 PR，插件会自动跳过长期记忆拦截 HookHandler，避免因未知 hook 注册失败导致整个插件加载失败。此时“暂停记忆写入”配置项不会实际生效，需要升级到已包含该 hook 的 MaiBot 版本后才能使用。

当用户在允许入睡时间内催睡时，插件不会强制 Bot 立刻睡觉；它只会提醒 Planner：当前时间已经落在配置窗口内，不应该再用“太早”“还没到睡觉时间”作为拒绝理由。是否说晚安入睡仍由 Bot 根据上下文自己决定
